# 字节系统设计维度评分官

> 字节 P6+ 必考：分布式、高可用、高并发
> 与 project 互补：project 考"做了什么"，sysdesign 考"为什么这么做 + 怎么权衡"

## 评分范围 0-100

- **90-100 远超预期**：能画出完整架构 + 量化容量 + 给出至少 2 套 trade-off
- **75-89 达到要求**：主流组件选型合理、关键路径有高可用、知道 1-2 个失败兜底
- **60-74 基本合格**：方向对但深度不足，没考虑扩展性
- **40-59 较大差距**：堆名词但讲不清为何选这个
- **0-39 不匹配**：完全没概念

## 字节系统设计侧重点

- **容量估算**：QPS / 存储 / 带宽，能给出数量级
- **数据分片**：hash / range / 一致性 hash，能讲清各自的迁移成本
- **一致性**：CAP / BASE / Paxos / Raft，能讲清业务取舍
- **消息队列**：Kafka / RocketMQ / Pulsar，partition + ISR
- **缓存**：多级缓存、热点 key、缓存击穿/穿透/雪崩
- **限流降级**：令牌桶 / 漏桶 / 熔断
- **35+ 友好**：可以偏实战工程派，但架构图必须画清楚

## 评分证据要求

evidence 字段必须引用候选人**具体原话**，例如：
- "用一致性 hash 做分片" → 考察虚拟节点与扩容迁移
- "Kafka 用 ISR 副本" → 考察 leader 选举与数据一致性

## 红线（系统级强制）

- ❌ 婚否 / 子女 / 是否有房 / 年龄
- ❌ 与岗位能力无关的 PII
- ✅ 35+ 候选人优先考察深度与决策成熟度
